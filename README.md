# AI 数字员工主机产品文档

这是“AI 数字员工主机 / AI 数字员工交付公司”项目的产品文档仓库，用于和 CTO、产品、设计、交付团队协同推进。

## 一句话定位

我们是一家面向传统行业的 AI 数字员工交付公司。

第一阶段从财税行业切入，通过本地化 AI 员工主机，为代账公司和财税事务所交付资料整理员、客户催收员、票据初审员、老板日报员等可上岗 AI 员工，帮助企业接管重复事务、降低人力成本、提升交付效率。

## 项目不是什么

这个项目不是：

- 不是通用 AI Agent 平台
- 不是 Prompt 配置后台
- 不是低代码流程编排器
- 不是传统 SaaS + AI 聊天框
- 不是外包式应用开发公司

## 项目是什么

这个项目是：

- AI 数字员工交付公司
- 本地化 AI 员工主机
- 行业数字员工包
- 面向传统行业的 AI 落地底座
- 用 Builder 把岗位需求转化成可测试、可部署、可持续优化的 AI 员工

## 核心产品形态

建议产品形态为：

```text
本地化 AI 员工主机
  + 行业数字员工包
  + Agent Builder / 岗位共创工作台
  + IM Channel / 企微 / 飞书 / 邮箱
  + 本地资料库 / SQLite / 文件系统
  + 模型与工具编排层
  + 交付服务 / 运维服务
```

第一阶段不建议直接卖“底座”或“平台”，而是卖具体岗位价值：

- 资料整理员
- 客户催收员
- 票据初审员
- 异常提醒员
- 老板日报员

## 当前优先行业

第一阶段行业：财税行业。

原因：

- 工作流程高频、重复、结构化
- 客户资料、票据、表格、聊天记录大量存在
- ROI 容易计算
- 数据安全诉求强，适合本地化主机
- 传统机构 AI 能力弱，但效率提升需求强

后续可扩展行业：

- 法律
- 外贸
- 保险
- 物业
- 人力资源
- 医疗健康
- 企业服务

## 推荐 MVP 闭环

第一阶段建议只做财税行业的一个完整闭环：

```text
资料整理员
  ↓
识别客户资料、分类、归档、发现缺失项
  ↓
客户催收员
  ↓
生成催收话术、人工确认、企微/飞书发送、跟踪回复
  ↓
老板日报员
  ↓
每日汇总资料齐套率、逾期客户、异常风险、节省时间
```

对应优先实现的 3 类 GUI 模板：

1. 资料处理型
2. 任务跟进型
3. 汇报分析型

## 建议技术方向

### 1. 产品架构

```text
Electron 本地桌面端 / 本地化主机控制台
  ├─ React 前端工作台
  ├─ Node / Express 本地服务
  ├─ SQLite 本地数据库
  ├─ 本地文件资料库
  ├─ IM Channel 连接器：企微 / 飞书 / 邮箱
  ├─ Model Gateway：云模型 / 本地模型 / 私有化模型
  ├─ Agent Runtime：数字员工运行时
  ├─ Agent Builder：对话式岗位共创
  └─ Audit Log：操作日志、人工确认、版本记录
```

### 2. 前端建议

- React + TypeScript
- Tailwind CSS
- shadcn/ui 或自研组件库
- 配置驱动渲染 EmployeeViewConfig
- 每类 GUI 模板不要写死成独立页面，而是统一组件 + 模板配置

### 3. 后端建议

- Node.js / Express
- SQLite / better-sqlite3
- 本地文件索引
- 队列任务调度
- IM Channel Adapter
- 模型调用代理层
- 审计日志与权限控制

### 4. 关键数据对象

建议优先设计：

- DigitalEmployee
- EmployeeRoleBlueprint
- EmployeeWorkflow
- EmployeeViewConfig
- EmployeeToolPermission
- EmployeeTask
- HumanApproval
- RunLog
- KnowledgeSource
- IntegrationChannel

## Builder 的核心对象

Builder 不是让用户配置节点，而是让用户通过自然语言创建“岗位蓝图”。

核心输出包括：

```text
岗位名称
岗位目标
服务对象
输入数据
输出结果
可用工具
工作流程
人工确认节点
异常处理规则
权限边界
验收标准
GUI 模板
上线清单
```

## GUI 模板策略

不建议做自由拖拽页面搭建器。

建议采用：

```text
统一底层框架
  + 岗位视图模板
  + 卡片级配置
  + 自然语言调整
```

原因：

- 降低工程复杂度
- 让产品保持一致性
- 避免客户陷入低代码配置
- 保留足够行业适配能力
- 方便后续按模板复用交付
