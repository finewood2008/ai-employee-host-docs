# 数字员工 GUI 模板库

所属项目：AI 数字员工主机

版本：v0.1

目标：定义一组“相对固定、可配置”的数字员工 GUI 模板，用来覆盖绝大多数行业数字员工场景。

核心原则：

> GUI 不做自由化定制，而做模板化可配置。

也就是说：

- 底层框架统一；
- 根据岗位类型选择工作视图模板；
- 每个模板由卡片、表格、动作区、日志区组成；
- Builder 通过自然语言调整卡片顺序、字段、指标、动作按钮和高亮规则；
- 不让客户自由拖拽页面或编辑底层组件。

---

## 一、统一底层框架

所有数字员工页面都建议统一成一个基础框架。

```text
┌─────────────────────────────────────────────────────────────┐
│ 员工身份栏：头像 / 名称 / 岗位 / 状态 / 最近运行 / 上线版本     │
├─────────────────────────────────────────────────────────────┤
│ 今日概览：关键指标 / 当前待办 / 异常数量 / AI 建议               │
├───────────────┬─────────────────────────────┬───────────────┤
│ 左侧列表区     │ 中间主工作区                  │ 右侧详情/动作区 │
│ 任务/客户/文件 │ 当前对象详情 / AI 判断 / 内容预览 │ 确认 / 操作 / 日志│
├───────────────┴─────────────────────────────┴───────────────┤
│ 底部：运行日志 / 版本记录 / 优化入口                           │
└─────────────────────────────────────────────────────────────┘
```

基础模块：

1. 员工身份栏
   - 员工头像
   - 员工名称
   - 岗位类型
   - 当前状态
   - 服务对象
   - 最近运行时间
   - 当前版本
   - 优化入口

2. 今日概览
   - 今日处理数量
   - 待人工确认数量
   - 异常数量
   - 节省时间
   - 关键风险
   - AI 今日建议

3. 左侧列表区
   - 任务列表
   - 客户列表
   - 文件列表
   - 风险列表
   - 报告列表

4. 中间主工作区
   - 当前任务详情
   - 文件预览
   - AI 识别结果
   - AI 判断依据
   - 生成内容
   - 当前流程状态

5. 右侧详情/动作区
   - 可执行动作
   - 人工确认
   - 下一步建议
   - 相关记录
   - 权限提示
   - 异常处理

6. 底部辅助区
   - 运行日志
   - 操作记录
   - 版本记录
   - 反馈优化

## 二、模板总览

建议第一阶段沉淀 7 种 GUI 模板，基本可以覆盖绝大多数数字员工场景。

1. 任务跟进型
2. 资料处理型
3. 审核风控型
4. 沟通话术型
5. 汇报分析型
6. 流程协同型
7. 知识问答型

它们分别覆盖：

| 模板 | 适合岗位 | 核心工作 |
| --- | --- | --- |
| 任务跟进型 | 客户催收员、销售跟进员、续保提醒员 | 盯人、盯事、盯进度 |
| 资料处理型 | 资料整理员、卷宗整理员、合同归档员 | 收资料、识别、分类、归档 |
| 审核风控型 | 票据初审员、合同审查员、理赔初审员 | 检查异常、识别风险、给结论 |
| 沟通话术型 | 客服回复员、客户催收员、销售助理 | 生成话术、确认、发送、跟踪回复 |
| 汇报分析型 | 老板日报员、经营分析员、客户健康监控员 | 汇总、分析、生成报告 |
| 流程协同型 | 项目协调员、申报推进员、交付调度员 | 多步骤流程推进、跨角色协作 |
| 知识问答型 | 政策顾问、法务助手、内部知识助理 | 基于知识库回答、引用资料、给建议 |

# 模板 1：任务跟进型

## 1. 适用场景

适合“盯进度、盯客户、盯任务”的数字员工。

典型岗位：

- 客户催收员
- 销售跟进员
- 续保提醒员
- 报修跟进员
- 线索跟进员
- 订单异常跟进员
- 逾期任务提醒员

典型行业：

- 财税
- 外贸
- 保险
- 物业
- 教育培训
- 医美
- 企业服务

## 2. 页面结构

```text
┌─────────────────────────────────────────────────────────────┐
│ 客户催收员 / 运行中 / 今日已跟进 23 个客户 / 待确认 8 条       │
├─────────────────────────────────────────────────────────────┤
│ 今日概览：待跟进 18｜已完成 23｜逾期 5｜需升级 2｜节省 3.5h    │
├───────────────┬─────────────────────────────┬───────────────┤
│ 跟进对象列表   │ 当前客户跟进详情              │ 下一步动作       │
│ - 待跟进       │ - 客户资料状态                │ - 生成话术       │
│ - 逾期未回复   │ - 最近沟通记录                │ - 创建任务       │
│ - 重点客户     │ - AI 判断                     │ - 提醒主管       │
│ - 已完成       │ - 缺失项/待办                 │ - 标记完成       │
├───────────────┴─────────────────────────────┴───────────────┤
│ 跟进日志 / 客户回复 / 版本优化                                │
└─────────────────────────────────────────────────────────────┘
```

## 3. 核心卡片

顶部概览卡：

- 今日待跟进
- 今日已完成
- 逾期未回复
- 需要人工确认
- 需要升级提醒
- 预计节省时间

左侧列表：

- 全部对象
- 待跟进
- 已跟进
- 逾期未回复
- 重点客户
- 需要主管介入

中间详情：

- 客户/任务基本信息
- 当前状态
- 上次跟进时间
- 最近沟通记录
- AI 判断结果
- 缺失事项/下一步事项
- 推荐处理方式

右侧动作：

- 生成跟进话术
- 创建确认任务
- 发送提醒
- 提醒负责人
- 标记已处理
- 暂缓跟进
- 加入重点关注

底部日志：

- AI 什么时候检查
- 生成了什么建议
- 人类是否确认
- 是否发送
- 客户是否回复

## 4. 可配置项

Builder 可通过自然语言配置：

- 跟进对象类型
- 跟进触发时间
- 逾期多久升级
- 跟进话术风格
- 是否发送前人工确认
- 重点对象置顶规则
- 列表字段
- 今日概览指标
- 日报内容

## 5. 典型自然语言调整

- “把逾期 2 天没回复的客户放到最上面。”
- “重点客户的催收话术温和一点。”
- “跟进前必须让负责人确认。”
- “日报里加上本周仍未回复的客户。”
- “不要显示节省时间，改成显示响应率。”

## 三、模板组合关系

现实中的数字员工可能不只属于一种模板。

例如：

### 客户催收员

主模板：任务跟进型

辅模板：沟通话术型

原因：既要盯客户资料进度，也要生成催收话术。

### 资料整理员

主模板：资料处理型

辅模板：流程协同型

原因：既要处理资料，也要把异常资料推给会计确认。

### 票据初审员

主模板：审核风控型

辅模板：资料处理型

原因：既要识别票据信息，也要判断风险。

### 老板日报员

主模板：汇报分析型

辅模板：知识问答型

原因：既要生成日报，也要回答老板追问。

### 合同审查助理

主模板：审核风控型

辅模板：知识问答型 + 资料处理型

原因：既要看合同文本，也要引用条款和案例。

因此，系统可以支持：

```ts
interface EmployeeViewConfig {
  primaryTemplate: EmployeeGuiTemplate;
  secondaryTemplates?: EmployeeGuiTemplate[];
}
```

但第一版可以先只实现主模板。

## 五、统一配置模型

前端不应该为每个员工生成一套新页面。

应该使用统一组件 + 配置渲染。

```ts
type EmployeeGuiTemplate =
  | 'task_followup'
  | 'document_processing'
  | 'review_risk'
  | 'communication'
  | 'report_analysis'
  | 'process_coordination'
  | 'knowledge_qa';

interface EmployeeViewConfig {
  employeeId: string;
  primaryTemplate: EmployeeGuiTemplate;
  secondaryTemplates?: EmployeeGuiTemplate[];
  title: string;
  subtitle: string;
  overviewCards: OverviewCardConfig[];
  leftPanel: PanelConfig;
  mainPanel: PanelConfig;
  rightPanel: PanelConfig;
  bottomPanel?: PanelConfig;
  actions: ActionConfig[];
  highlightRules: HighlightRule[];
}

interface OverviewCardConfig {
  id: string;
  label: string;
  metricKey: string;
  unit?: string;
  priority: number;
  visible: boolean;
}

interface PanelConfig {
  title: string;
  components: ViewComponentConfig[];
}

interface ViewComponentConfig {
  id: string;
  type:
    | 'task_list'
    | 'customer_list'
    | 'file_queue'
    | 'document_preview'
    | 'ai_extraction'
    | 'risk_evidence'
    | 'message_editor'
    | 'approval_queue'
    | 'report_preview'
    | 'workflow_board'
    | 'qa_thread'
    | 'citation_list'
    | 'activity_log'
    | 'metric_grid';
  title: string;
  visible: boolean;
  priority: number;
  config: Record<string, any>;
}

interface ActionConfig {
  id: string;
  label: string;
  type: 'primary' | 'secondary' | 'danger';
  requiresConfirmation: boolean;
  permissionKey: string;
}

interface HighlightRule {
  id: string;
  condition: string;
  style: 'red' | 'yellow' | 'green' | 'blue' | 'bold' | 'pin_top';
  description: string;
}
```

## 六、第一阶段建议优先实现的 3 个模板

如果资源有限，不建议 7 个模板一次做完。

第一阶段建议先实现 3 个：

### 1. 任务跟进型

覆盖客户催收员。

这是第一版 Agent Builder 的核心演示场景。

### 2. 资料处理型

覆盖资料整理员。

财税行业高频刚需，和本地资料库、文件识别强相关。

### 3. 汇报分析型

覆盖老板日报员。

最容易让老板感知价值，也适合做演示闭环。

这三个模板可以组成财税 MVP 的完整闭环：

> 资料整理员收资料 → 客户催收员催缺失 → 老板日报员汇报结果

第二阶段再补：

- 审核风控型：票据初审员
- 沟通话术型：客户沟通/销售跟进
- 流程协同型：申报推进
- 知识问答型：政策顾问

## 七、总结

数字员工 GUI 不应该做成完全自由定制。

最合理的方式是：

> 固定基础框架 + 7 类岗位视图模板 + 卡片级配置 + 自然语言调整。

7 类模板包括：

1. 任务跟进型
2. 资料处理型
3. 审核风控型
4. 沟通话术型
5. 汇报分析型
6. 流程协同型
7. 知识问答型

这 7 类基本可以覆盖大多数行业数字员工：

- 盯任务；
- 处理资料；
- 审核风险；
- 对外沟通；
- 汇报分析；
- 推进流程；
- 回答问题。

第一阶段建议优先实现：

1. 任务跟进型
2. 资料处理型
3. 汇报分析型

它们可以支撑财税行业 MVP 的核心闭环。
