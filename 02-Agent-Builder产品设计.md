# Agent Builder 产品设计讨论稿

所属项目：AI 数字员工主机

版本：v0.1

---

## 一、核心判断

Agent Builder 这个板块，不能做成“低代码流程编排器”，也不能做成“Prompt 配置后台”。

如果这样做，它会很像 Dify、Coze、FastGPT、Flowise、n8n、Zapier、LangGraph Studio 这类工具，客户会立刻进入“我不会配、我也不想配”的状态。

对我们这个产品来说，Agent Builder 的核心应该是：

> 用对话式共创的方式，把一个真实岗位设计、生成、测试、部署成一个可上岗的 AI 数字员工。

也就是说，它不是“开发者 Builder”，而是“岗位共创 Builder”。

---

## 二、Agent Builder 在整个产品里的位置

整个产品大概有三层。

### 第一层：AI 员工主机

这是底座。

负责：

- 本地数据；
- 文件库；
- 模型网关；
- 工具连接器；
- 任务队列；
- 权限；
- 日志；
- 审计；
- 工作流运行；
- 备份恢复。

### 第二层：AI 数字员工团队

这是客户真正感知到的产品。

比如财税行业：

- 资料整理员；
- 客户催收员；
- 票据初审员；
- 异常提醒员；
- 老板日报员。

客户买的是这些员工。

### 第三层：Agent Builder

这是创建、调整、测试、上线 AI 员工的工作台。

但注意：

它不是给客户从零搭复杂 Agent 的地方。

它更像一个“AI 岗位设计师”。

客户只需要说：

- “我想让 AI 帮我催客户每月交资料”；
- “我希望它每天检查哪些客户还没交发票”；
- “催收话术要客气一点，别太强硬”；
- “发送前先让我确认”；
- “超过三天没回复就提醒主管”。

Builder 要把这些自然语言需求转成：

- 岗位职责；
- 输入来源；
- 工具权限；
- 工作流程；
- 判断规则；
- 话术风格；
- 人工确认节点；
- 异常处理；
- 验收标准；
- 测试用例；
- 上线配置。

所以 Agent Builder 的定位应该是：

> 面向业务用户的 AI 数字员工共创工作台。

不是：

> 技术人员配置 Agent 的工具。

---

## 三、Agent Builder 的一句话定义

### 标准定义

Agent Builder 是一个对话式数字员工设计工作台，帮助用户把一个真实岗位拆解为职责、数据、工具、流程、权限和验收标准，并自动生成可测试、可部署、可持续优化的 AI 员工。

### 口语版

你不用学怎么搭 Agent，只需要和 Builder 聊清楚“这个员工要干什么”，它会帮你生成岗位说明书、工作流、工具权限和测试方案，确认后就能上线。

### 产品化表达

从岗位想法到 AI 员工上线的共创系统。

---

## 四、Agent Builder 不是“表单”，而是“共创”

传统 Builder 会让用户填：

- Agent 名称；
- 系统提示词；
- 知识库；
- 工具；
- 触发器；
- 工作流；
- 变量；
- API；
- 权限；
- 输出格式。

这对中小企业老板和财税公司负责人来说太复杂。

他们不想配置。

他们只想描述业务：

> “每个月 5 号之后，如果客户还没交发票，就提醒会计催客户。催收话术要根据客户类型不同，重要客户语气柔和一点，长期拖延客户可以强一点。发送前要给会计确认。”

所以 Builder 应该问的是业务问题，而不是技术问题：

- 这个员工替谁干活？
- 它每天要检查什么？
- 它从哪里拿资料？
- 它什么时候开始工作？
- 它发现问题后通知谁？
- 哪些动作需要人工确认？
- 哪些动作可以自动执行？
- 出错时找谁？
- 它的工作结果怎么验收？

也就是说：

> 不要让用户配置 Agent。让 Builder 采访用户，然后自动生成 Agent。

## 五、Agent Builder 的核心用户

建议分两类用户，但界面第一阶段主要服务第一类。

### 1. 内部交付团队

第一阶段最重要。

因为客户不会自己 build agent。

内部交付团队用 Builder 来：

- 快速理解客户流程；
- 生成员工岗位说明书；
- 生成工作流；
- 生成测试用例；
- 生成部署配置；
- 沉淀行业模板。

这是效率工具。

它帮助我们把交付成本降下来。

### 2. 客户侧业务负责人

第二阶段开放。

比如财税公司老板、主管、会计负责人。

他们可以用 Builder 做轻量调整：

- 新增一个员工；
- 调整话术；
- 增加确认节点；
- 修改日报内容；
- 增加一个触发条件；
- 查看员工表现；
- 发起优化请求。

但他们不应该看到复杂技术配置。

所以产品上可以分成：

- 内部专业模式；
- 客户业务模式。

第一版建议先做：

> 业务模式为主，专业配置隐藏在右侧或高级设置里。

---

## 六、核心对象：不是 Agent，而是“岗位”

不要让用户创建“Agent”。

让用户创建“AI 员工”或者“岗位”。

一个 AI 员工应包含以下内容。

### 1. 岗位身份

例如：

- 资料整理员；
- 客户催收员；
- 票据初审员；
- 销售跟进员；
- 老板日报员。

### 2. 岗位目标

例如：

> 每月自动检查客户资料是否齐全，并协助会计完成催收。

### 3. 服务对象

它为谁工作？

- 老板；
- 主管；
- 会计；
- 销售；
- 客服；
- 客户成功。

### 4. 输入来源

它从哪里获得信息？

- 企业微信聊天；
- 飞书群；
- 本地文件夹；
- 邮箱；
- 客户资料库；
- 发票文件；
- Excel 表格；
- PDF；
- 业务系统。

### 5. 工具权限

它能做什么？

- 读文件；
- 识别发票；
- 查询客户资料；
- 创建任务；
- 生成话术；
- 发送企微消息；
- 写入客户档案；
- 生成日报；
- 提醒负责人。

### 6. 工作流程

它怎么工作？

- 定时检查；
- 收到资料后触发；
- 识别资料；
- 匹配客户；
- 判断缺失；
- 生成建议；
- 等待确认；
- 发送消息；
- 记录结果；
- 异常上报。

### 7. 人工确认节点

哪些地方必须人确认？

- 发送客户消息前；
- 归档异常资料前；
- 标记风险客户前；
- 生成正式报告前；
- 调用外部系统前。

### 8. 话术风格

例如：

- 专业；
- 温和；
- 简洁；
- 强提醒；
- 老板口吻；
- 客服口吻；
- 财税顾问口吻。

### 9. 异常处理

例如：

- 无法识别客户；
- 资料模糊；
- 客户名称冲突；
- 发票重复；
- 金额异常；
- 客户未回复；
- 任务超时。

### 10. 验收标准

怎么判断这个员工干得好？

- 资料归档准确率；
- 缺失识别准确率；
- 催收响应率；
- 人工修改率；
- 任务逾期下降；
- 节省人工时间；
- 用户满意度。

这才是 Agent Builder 真正要构建的东西。

## 七、Agent Builder 的产品形态

建议采用“三栏工作台”，但不是传统配置台。

### 左侧：对话共创区

用户和 Builder 对话。

用途：

- 描述需求；
- 回答 Builder 提问；
- 确认方案；
- 修改员工设定；
- 查看 Builder 建议。

### 中间：员工蓝图画布

实时生成 AI 员工蓝图。

展示：

- 岗位卡片；
- 数据来源；
- 工作流程；
- 工具权限；
- 人工确认节点；
- 异常分支；
- 输出结果。

它不是复杂流程图，而是业务看得懂的“岗位工作图”。

### 右侧：属性与测试面板

展示当前选中节点的细节。

用途：

- 查看岗位说明；
- 调整规则；
- 查看工具权限；
- 运行测试样本；
- 查看日志；
- 查看验收指标。

### 视觉结构

```text
┌──────────────────┬──────────────────────────────┬──────────────────┐
│  Builder 对话区   │        员工蓝图画布             │   属性 / 测试面板  │
│                  │                              │                  │
│  “我想做一个...”  │  资料来源 → 判断规则 → 动作     │  节点说明         │
│  Builder 提问     │  人工确认 → 输出结果            │  权限             │
│  方案确认         │                              │  测试样本         │
│                  │                              │  验收标准         │
│  输入框           │                              │  上线按钮         │
└──────────────────┴──────────────────────────────┴──────────────────┘
```

这个形态非常适合我们的产品。

因为它既是 AI Native，又保留了“可视化可控”的感觉。

## 八、核心流程：从一句话到员工上线

Agent Builder 的主流程应该是 6 步。

### 第 1 步：创建岗位

用户输入：

> 我想做一个客户催收员，帮会计每个月催客户交资料。

Builder 输出：

> 我理解你要创建一个“客户催收员”。
>
> 它的目标是：每月检查客户资料是否齐全，并协助会计催收缺失资料。
>
> 我需要确认 5 个问题：
>
> 1. 它服务谁？会计、主管还是老板？
> 2. 它从哪里判断客户资料是否齐？
> 3. 它什么时候检查？
> 4. 催收消息是否需要人工确认？
> 5. 催收失败几天后升级提醒谁？

### 第 2 步：业务访谈

Builder 用对话方式补齐关键信息。

注意：不能一次问 20 个问题。

要分轮问，每次 3-5 个。

并且给默认建议。

例如：

> 财税行业通常建议发送客户消息前先由会计确认。是否采用这个默认规则？

这样用户只需要确认。

### 第 3 步：生成员工蓝图

Builder 生成：

- 岗位说明；
- 数据来源；
- 流程图；
- 工具权限；
- 话术风格；
- 确认节点；
- 异常处理；
- 验收标准。

中间画布出现一张员工蓝图。

例如：

#### 客户催收员

触发：

- 每月 5 日 09:00 自动检查；
- 收到新资料后重新计算缺失状态。

输入：

- 客户资料库；
- 企业微信文件；
- 本地文件夹；
- 上月资料清单。

判断：

- 识别每个客户本月应提交资料；
- 对比已归档资料；
- 标记缺失项。

动作：

- 生成催收话术；
- 创建会计确认任务；
- 确认后发送客户；
- 记录催收状态。

异常：

- 客户名称不匹配 → 推送会计确认；
- 资料模糊 → 标记待人工识别；
- 3 天未回复 → 提醒主管。

### 第 4 步：测试运行

这是非常关键的一步。

不能直接上线。

Builder 应该要求用户提供或选择测试样本。

例如：

- 选择 3 个客户；
- 选择 10 份资料；
- 模拟本月 5 号检查；
- 生成催收话术；
- 查看任务生成结果。

测试结果展示：

- 识别了多少资料；
- 匹配了哪些客户；
- 哪些资料缺失；
- 生成了哪些消息；
- 哪些地方需要人工确认；
- 潜在风险是什么。

### 第 5 步：上线确认

上线前给用户一个清单：

- 员工名称：客户催收员；
- 服务对象：会计；
- 触发时间：每月 5 日 09:00；
- 发送客户消息前：需要人工确认；
- 异常升级：3 天未回复提醒主管；
- 权限：读取客户资料、生成任务、生成话术、发送企微消息；
- 禁止：自动修改客户档案、自动删除资料、自动发送未经确认消息。

用户点击：

> 确认上线

### 第 6 步：持续优化

上线后不是结束。

Builder 应该变成员工优化入口。

用户可以说：

- “催收话术太硬了，温和一点。”
- “老板日报里加上逾期客户名单。”
- “客户三天没回复，改成两天提醒。”
- “资料整理员经常把 A 公司和 B 公司混淆，帮我优化。”
- “这个员工上周帮我们节省了多少时间？”

Builder 自动调整员工配置，并记录版本。

## 九、Agent Builder 的核心页面模块

### 1. 顶部：员工创建状态栏

显示当前阶段：

> 需求描述 → 业务访谈 → 蓝图生成 → 测试运行 → 上线确认 → 持续优化

不要像流程表单，要像项目进度。

### 2. 左侧：Builder 对话

特点：

- 主入口；
- 自然语言；
- Builder 主动提问；
- 用户确认建议；
- 支持上传样本资料；
- 支持引用已有员工模板。

输入框提示：

> 描述你想创建的 AI 员工，例如：帮我每月催客户交资料

快捷按钮：

- 从模板创建；
- 上传流程说明；
- 导入现有岗位；
- 优化已有员工。

### 3. 中间：员工蓝图画布

不是 Figma 式自由画布，而是结构化蓝图。

包含 6 类节点：

- 岗位目标；
- 数据来源；
- 判断规则；
- 执行动作；
- 人工确认；
- 输出结果。

可以用横向流：

> 输入 → 理解 → 判断 → 执行 → 确认 → 输出

也可以分区：

- 岗位设定；
- 资料来源；
- 工作流程；
- 权限边界；
- 异常处理；
- 验收指标。

第一版建议用“分区卡片”，不要做复杂拖拽流程图。

因为复杂流程图开发成本高，也容易让用户焦虑。

### 4. 右侧：属性 / 测试 / 版本

右侧可以三个 tab。

#### 属性

显示选中节点的业务解释和参数。

例如选中“发送催收消息”：

- 动作名称：生成并发送催收话术；
- 触发条件：客户资料缺失；
- 确认要求：发送前需要会计确认；
- 可用渠道：企业微信；
- 语气风格：专业、温和；
- 失败处理：客户 3 天未回复提醒主管。

#### 测试

可以运行样本。

显示：

- 输入样本；
- AI 判断；
- 生成结果；
- 人工确认点；
- 错误/风险提示。

#### 版本

显示员工配置历史。

例如：

- v1.0 创建客户催收员；
- v1.1 催收话术改为温和；
- v1.2 未回复升级从 3 天改为 2 天；
- v1.3 增加老板日报摘要。

## 十、第一版不要做什么

这点很重要。

### 1. 不做自由拖拽流程编排

不要做 n8n/Flowise 那种节点画布。

原因：

- 开发复杂；
- 用户难懂；
- 容易偏工程工具；
- 不符合“业务共创”的定位。

### 2. 不暴露 Prompt 编辑器

不要让客户看到大段 system prompt。

他们不该改 prompt。

应该看到的是：

- 职责；
- 规则；
- 话术风格；
- 权限；
- 确认节点；
- 验收标准。

底层 prompt 可以自动生成，但不要作为主界面。

### 3. 不让客户自己配 API

工具授权和连接器应该是系统级能力。

客户只需要知道：

- 这个员工能不能读取客户资料？
- 能不能发企微？
- 能不能创建任务？

不要让他填 webhook、token、schema。

### 4. 不做通用 Agent 市场

第一版不要做“什么都能创建”。

应该聚焦财税员工。

例如：

- 资料整理员；
- 客户催收员；
- 票据初审员；
- 异常提醒员；
- 老板日报员。

从这些模板开始。

### 5. 不做完全自动化

第一版必须强调人工确认。

尤其是：

- 发送客户消息；
- 写入正式档案；
- 标记风险；
- 生成正式报告；
- 调用外部系统。

---

## 十一、关键设计原则

### 1. 岗位优先，不是 Agent 优先

用户创建的是“员工”，不是“Agent”。

### 2. 对话共创，不是表单配置

用户说业务，Builder 转成结构。

### 3. 业务语言，不是技术语言

少说 prompt、tool、workflow、schema，多说职责、资料、动作、确认、汇报。

### 4. 默认建议，不要空白配置

Builder 应该主动给行业默认方案。

### 5. 可视化蓝图，不是复杂流程图

让用户看懂员工如何工作，但不要让用户拖一堆节点。

### 6. 先测试，再上线

每个员工上线前必须跑样本测试。

### 7. 人工确认优先

第一版避免高风险全自动动作。

### 8. 上线后持续优化

Builder 不只是创建工具，也是员工优化入口。

## 十二、具体示例：创建“客户催收员”

### 用户

> 我想做一个客户催收员，帮我们每个月催客户交资料。

### Builder

好的，我会帮你创建一个“客户催收员”。

它的默认目标是：

> 每月自动检查客户资料是否齐全，发现缺失后生成催收话术，经会计确认后通过企业微信发给客户，并记录催收状态。

我建议采用财税行业默认规则：

1. 每月 5 日上午 9 点自动检查；
2. 发送客户消息前必须由会计确认；
3. 客户 3 天未回复，提醒主管；
4. 所有催收记录自动写入客户档案；
5. 每天下午 6 点生成催收进度摘要。

你要使用这个默认方案吗？

### 用户

> 可以，但我们一般每月 3 号开始催。

### Builder

已调整为每月 3 日上午 9 点检查。

接下来我需要确认资料范围。

客户每月必须提交哪些资料？

默认清单是：

- 发票；
- 银行回单；
- 工资表；
- 社保公积金资料；
- 其他合同或费用凭证。

是否需要增加或删除？

### 用户

> 先按这个来。

### Builder

好的，我已经生成客户催收员蓝图。

它将使用以下数据来源：

- 客户资料库；
- 企业微信聊天文件；
- 本地资料文件夹；
- 历史月份资料清单。

它拥有以下权限：

- 读取客户资料；
- 判断资料缺失；
- 生成催收话术；
- 创建会计确认任务；
- 经确认后发送企业微信消息；
- 写入催收记录。

它不允许：

- 未经确认直接发送客户消息；
- 删除客户资料；
- 修改客户基础档案；
- 标记客户风险等级。

现在可以选择 3 个客户进行测试运行。

## 十四、Agent Builder 和员工运行态的关系

这是产品里很容易混淆的地方。

Agent Builder 是创建和优化员工的地方。

员工运行态是员工真正干活的地方。

比如：

1. Builder 里创建“客户催收员”；
2. 上线后，这个员工出现在“AI 员工团队”；
3. 每天运行的结果出现在“任务池 / 日报 / 客户档案”；
4. 如果用户发现问题，再回 Builder 优化。

所以产品路径是：

> Builder 创建 → 员工上线 → 工作台运行 → 发现问题 → Builder 优化 → 新版本上线

不要让 Builder 变成员工日常工作页面。

它是设计室，不是车间。

## 十五、第一版 MVP 建议

如果要做 Agent Builder MVP，建议只做一个场景：

> 创建“客户催收员”

为什么？

因为它最能体现 Builder 的价值：

- 有明确岗位；
- 有资料来源；
- 有判断规则；
- 有话术生成；
- 有人工确认；
- 有外部动作；
- 有异常升级；
- 有日报输出。

MVP 功能：

1. 从模板开始创建客户催收员；
2. 对话式访谈；
3. 自动生成员工蓝图；
4. 显示权限和确认节点；
5. 运行模拟测试；
6. 生成上线清单；
7. 保存为员工草稿。

第一版可以不真的接企微，不真的执行，只要把产品体验跑通。

但如果要有演示说服力，至少要有模拟数据和模拟运行结果。

## 十六、最终产品感

它应该给人的感觉不是：

> 我在配置一个复杂系统。

而是：

> 我在和一个很懂业务的 AI 产品经理一起设计一个新员工。

它应该像：

- 面试一个员工；
- 写岗位说明书；
- 制定工作流程；
- 安排权限；
- 试岗；
- 转正上线；
- 持续绩效优化。

这个类比非常好。

因此页面文案也可以围绕“员工入职”来设计：

- 创建岗位；
- 岗位访谈；
- 员工蓝图；
- 试岗测试；
- 上线确认；
- 绩效优化。

而不是：

- 创建 Agent；
- 配置 Prompt；
- 绑定 Tool；
- 编排 Workflow；
- Deploy。

这就是我们和传统 Agent 平台的区别。

## 十七、初步产品建议

### 产品定位

Agent Builder 可以定为：

> AI 数字员工入职设计台

### 核心体验

> 像招聘和培训一个新员工一样，创建一个 AI 员工。

### 核心流程

> 岗位描述 → 业务访谈 → 员工蓝图 → 试岗测试 → 上线确认 → 持续优化

### 核心界面

- 左侧对话共创；
- 中间员工蓝图；
- 右侧属性测试。

### 第一版场景

财税客户催收员。

### 第一版不要做

- 自由拖拽流程图；
- Prompt 编辑器；
- API 配置；
- 通用 Agent 市场；
- 高风险自动执行。

## 十八、下一步需要继续讨论的问题

在进入 UI 原型前，需要继续明确 5 件事。

### 1. Agent Builder 的用户到底是谁？

- 内部交付人员为主？
- 客户老板为主？
- 客户主管为主？
- 还是三者都要兼容？

建议：第一阶段内部交付人员为主，客户业务负责人可看懂、可参与确认。

### 2. Builder 第一版到底创建哪个员工？

- 客户催收员？
- 资料整理员？
- 老板日报员？
- 还是从模板库选多个？

建议：先做客户催收员。

### 3. Builder 的结果是什么？

- 仅生成蓝图？
- 生成可运行配置？
- 生成测试报告？
- 生成上线员工？
- 还是都要有？

建议：MVP 生成“可运行配置 + 测试报告 + 上线清单”。

### 4. 画布要做到什么程度？

- 结构化卡片？
- 流程图？
- 可拖拽节点？
- 可编辑蓝图？

建议：先做结构化卡片 + 简单流程，不做复杂拖拽。

### 5. 客户能不能看到高级配置？

建议：默认不看。

右侧可以有“高级详情”，但不出现 prompt、API、schema 这类词。

## 十九、阶段性结论

Agent Builder 的本质不是“搭 Agent 的工具”，而是：

> 数字员工入职设计台。

它要帮助用户把一个真实岗位，通过对话共创的方式，拆解成：

- 职责；
- 数据；
- 工具；
- 流程；
- 权限；
- 确认节点；
- 异常处理；
- 验收标准。

然后生成一个可以测试、上线、持续优化的 AI 员工。

这就是我们和传统 Agent 平台最大的区别。
